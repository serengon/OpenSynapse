## Dónde estamos

**v0.1 MVP funcional, end-to-end probado.** Daemon `opensynapsed` que:
- Detecta devices Razer vía openrazer DBus.
- Carga scenes y device profiles desde `~/.config/opensynapse/`.
- Cambia automáticamente el RGB del Tartarus Pro al cambiar de ventana activa (X11/KDE).
- Bootstrap en primer arranque (genera `_aliases.toml` + scene fallback).

Pendiente de validación visual por el usuario: ejecutar el daemon mirando el Tartarus, confirmar que el switching realmente funciona (la receta está en `README.md` Quickstart y en el último mensaje de la sesión).

## Pregunta abierta — primera decisión de mañana

Usuario propuso armar **GUI**. Le ofrecí tres versiones y se fue a dormir antes de elegir:

- **A — Tray icon mínimo** (1-2 sesiones). Selector de scene + status. Mejor relación valor/esfuerzo, no bloquea otras features.
- **B — Editor visual de configs** (3-5 sesiones). Reemplaza editar TOML a mano. Útil pero TOML está bastante OK por ahora.
- **C — Synapse completa** (semanas). El sueño del roadmap; prematuro porque faltan features (macros, audio, importador) que la GUI tendría que mostrar.

Mi recomendación de cierre fue **A primero**, después evaluar B. C postergado.

Si el usuario elige cualquier opción de GUI, antes de codear hace falta decidir:
1. **Framework**: Qt/QML (lo que dicen los ADRs, alineado KDE), Slint (Rust-native), GTK4+relm4, Tauri (no para system app), Iced (immediate mode, menos nativo).
2. **Relación con el daemon**: GUI separada que habla por DBus/socket vs GUI integrada al binario del daemon.

## Si NO elige GUI mañana

Próximos no-GUI candidatos (en orden de utilidad):
- **Hot-reload de configs en el daemon** (SIGUSR1 → recarga). Quality of life real, ~30 min.
- **Spike #8 importador Polychromatic** — desbloquea adopción y es criterio de "done" del v0.1.
- **Spike #3 macros** — input-remapper o keyd. Define cuál antes de codear (decision aún sin tomar).
- **Soporte logo-only en Nari** — el adapter actual usa `razer.device.lighting.chroma` que el Nari no expone. Sumar `razer.device.lighting.logo` (`setLogoStatic`/`setLogoNone`/`setLogoBrightness`) y que `apply_lighting` haga fallback automático según qué interfaces existen.

## Loose ends técnicos (sin bloqueo)

- Errores de openrazer llegan con la traceback de Python entera; opcional limpiarlos en el adapter para logs más legibles.
- Discovery del adapter no cachea — cada `apply_lighting` re-enumera. Para v0.1 está OK; cuando haya hot-plug habrá que cachear con TTL (ya documentado en ADR 0004).
- Engine no tiene tests automatizados. Se pueden agregar con un mock LightingAdapter — solo lógica pura de matching y dedup.

## Hallazgos guardados en memoria del proyecto

- openrazer corre desde fork local del usuario (`~/GitHub/openrazer`, branch `add-nari-ultimate`); por eso el Nari aparece pero el `chroma` interface está limitado.
- openrazer DBus usa **camelCase** y la interface de batería es `razer.device.power` (no `razer.device.battery` como decía un research inicial). Errores remotos llegan como `MethodError`, no `FDO`.
- KDE+X11 emite **dos** PropertyNotify de `_NET_ACTIVE_WINDOW` por cada switch; dedup vive en el orchestrator (ya implementado).
